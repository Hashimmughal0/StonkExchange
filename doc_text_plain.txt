1. Project Overview
1.1 Purpose
StockDB Exchange is a database project that simulates a real-world stock exchange. The primary learning objective is to design and implement robust relational database schemas, advanced SQL queries, stored procedures, triggers, transactions, and indexing strategies — with a working frontend and backend tying it all together.
1.2 Project Goals
Design a normalized relational database schema covering users, stocks, orders, trades, wallets, and audit logs.
Implement all core exchange logic (order matching, settlement, portfolio tracking) at the database layer using stored procedures, triggers, and views.
Build a Node.js REST API that acts as a thin layer over the database — business logic lives in SQL, not application code.
Deliver a React frontend that visualizes the order book, trade history, portfolio, and real-time price charts.
Demonstrate advanced database concepts: ACID transactions, indexing, query optimization, and concurrency control.
1.3 Scope
In Scope:
User registration, authentication, and session management
Stock listings and price data
Order placement: Market, Limit, and Stop-Loss orders
Order book management and real-time matching engine (via DB triggers/procedures)
Trade execution, settlement, and wallet/balance management
Portfolio tracking and P&amp;L calculation
Admin panel: stock management, user overview, system stats
Audit logging for all critical operations
Out of Scope:
Real-time WebSocket market data feeds (mock/polling acceptable)
Payment gateway integration
Regulatory compliance (KYC/AML)
Mobile native apps

2.2 Architecture Principles
Database-First: All business logic (order matching, balance checks, trade settlement) is implemented as stored procedures and triggers.
Thin API: Node.js endpoints only validate HTTP input and call stored procedures — no business logic in JavaScript.
ACID Compliance: Every trade execution runs inside a database transaction to ensure atomicity and consistency.
Audit Everything: Triggers automatically log inserts/updates on critical tables to an audit log.

4.4 Order Matching Algorithm
The matching engine (match_orders stored procedure) implements price-time priority (FIFO):
When a new order is inserted, the AFTER INSERT trigger fires match_orders(stock_id).
match_orders fetches the best open buy limit order (highest limit_price) and best open sell limit order (lowest limit_price) for the stock.
If best_buy.limit_price &gt;= best_sell.limit_price, a match is possible. Execution price = sell order price (maker-taker model).
Trade quantity = MIN(buy.remaining_qty, sell.remaining_qty). execute_trade() is called atomically.
Both orders are updated: filled_quantity incremented, status set to &apos;partial&apos; or &apos;filled&apos;.
Loop continues until no more matches exist for the stock.
Market orders are matched first against any available opposite limit order regardless of price.

7. Security &amp; Auth Design
7.1 Authentication Flow
User submits credentials → API validates against users table (bcrypt.compare)
On success: generate signed JWT (user_id, role, expiry) → stored in sessions table
All protected routes: middleware verifies JWT signature and checks sessions table
Logout: deletes session record from sessions table → token invalidated server-side
7.2 Authorization
Role-based: &apos;user&apos; and &apos;admin&apos; roles enforced at API middleware layer
Row-level security: users can only view/cancel their own orders (WHERE user_id = $1 in all queries)
Admin endpoints protected by role check middleware
7.3 Data Security
Passwords stored as bcrypt hashes (cost factor 12) — never plaintext
All SQL queries use parameterized statements — zero SQL injection risk
Wallet mutations only via stored procedures — no direct table UPDATE from API
Audit log is append-only (INSERT only trigger) — no UPDATE or DELETE permitted

9. Non-Functional Requirements
Category
Requirement
Performance
Order book page must load within 1 second; matching procedure must complete within 100ms for typical order books
Data Integrity
No trade may execute without sufficient balance or holdings — enforced by stored procedures with CHECK constraints
Scalability
Schema must support at least 100,000 orders and 1,000,000 price history records without index redesign
Reliability
All trade executions must be fully atomic — partial execution failure must roll back entire transaction
Auditability
Every balance change and order state change must produce an audit_log entry with timestamp and actor
Security
No raw SQL from user input — all queries parameterized; JWT validation on every protected route
